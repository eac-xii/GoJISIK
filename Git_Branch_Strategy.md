# Git Branch 전략 이야기: 우리 팀에 맞는 흐름은 무엇일까?

Git을 쓰기 시작하면 누구나 한 번쯤 이런 고민을 하게 된다.  
“브랜치를 어떻게 나누는 게 맞지?”  
“feature, develop, main… 이게 꼭 다 필요한 걸까?”

사실 Git Branch 전략은 정답이 정해져 있지 않다.  
프로젝트의 규모, 팀의 숙련도, 배포 방식에 따라 **잘 맞는 전략**이 있을 뿐이다.  
이 글에서는 실무에서 자주 쓰이는 Git Branch 전략들을 하나씩 살펴보면서,  
각 전략이 어떤 배경에서 나왔고 어떤 상황에 어울리는지 자연스럽게 풀어본다.

## 왜 Git Branch 전략이 중요할까?

브랜치 전략이 없거나 흐릿하면 이런 일이 자주 생긴다.

- 누군가 작업한 코드가 갑자기 운영 환경에 섞여 들어간다
- 배포 직전에 어떤 커밋이 포함됐는지 아무도 정확히 모른다
- 긴급 버그 수정 하나 때문에 전체 개발 흐름이 꼬인다
- “이거 어느 브랜치에서 작업해야 해요?”라는 질문이 반복된다

브랜치 전략은 단순히 Git 규칙을 정하는 게 아니라,  
**팀의 협업 방식과 배포 철학을 코드로 표현하는 방법**에 가깝다.

## 1. Git Flow — 가장 교과서적인 방식

Git Flow는 가장 오래되고, 가장 많이 알려진 전략이다.  
그래서인지 처음 Git을 배울 때 가장 먼저 접하는 경우도 많다.

이 전략의 핵심은 한 가지다.  
**“개발 중인 코드와 실제로 배포되는 코드를 명확히 나누자.”**

### 어떤 식으로 흘러갈까?

보통 흐름은 이렇다.

- 운영에 나가는 코드는 `main`
- 다음 버전을 준비하는 개발 코드는 `develop`
- 새로운 기능은 `feature/*` 브랜치에서 작업
- 배포가 가까워지면 `release/*` 브랜치에서 마무리
- 운영 중 문제는 `hotfix/*`로 빠르게 대응

단계별로 역할이 딱 나뉘어 있어서,  
“지금 이 코드는 어느 상태인가?”를 브랜치 이름만 보고도 알 수 있다.

### 장점은?

- 구조가 명확해서 처음 보면 이해가 쉽다
- 릴리즈 단위 관리가 깔끔하다
- 운영 안정성이 매우 높다

### 아쉬운 점은?

- 브랜치가 많아 관리가 번거롭다
- 작은 수정에도 병합 과정이 늘어난다
- 빠른 배포가 중요한 프로젝트에는 다소 무겁다

### 이런 프로젝트에 잘 맞는다

- 중·대형 규모의 팀
- 릴리즈 주기가 분명한 서비스
- “안정성 > 속도”인 조직 문화

## 2. GitHub Flow — 최대한 단순하게

GitHub Flow는 Git Flow를 쓰다 보니 나온 현실적인 질문에서 시작됐다.  
“이렇게까지 복잡할 필요가 있을까?”

그래서 핵심만 남겼다.

- 기준은 오직 `main`
- 모든 작업은 `feature/*` 브랜치에서 시작
- 작업이 끝나면 Pull Request로 검토 후 `main`에 병합
- `main`은 항상 배포 가능한 상태

### 이 전략의 철학

GitHub Flow는 이렇게 말한다.  
**“배포 가능한 코드를 계속 유지하자.”**

릴리즈 브랜치나 develop 브랜치를 따로 두지 않고,  
리뷰와 자동화를 통해 안정성을 확보한다.

### 장점은?

- 구조가 정말 단순하다
- 브랜치 고민이 줄어든다
- 빠른 개발과 배포에 유리하다

### 단점은?

- 릴리즈 단위 관리가 어렵다
- QA나 스테이징 환경이 분리되지 않으면 위험하다

### 이런 프로젝트에 잘 맞는다

- 소규모 팀
- 웹 서비스, SaaS
- 잦은 배포가 일상인 프로젝트

## 3. GitLab Flow — 환경을 기준으로 생각하기

GitLab Flow는 GitHub Flow를 기반으로 하지만,  
여기에 “배포 환경”이라는 개념을 더한다.

“브랜치는 코드의 상태뿐 아니라,  
어디에 배포되느냐도 표현해야 하지 않을까?”라는 고민에서 나왔다.

### 어떤 식으로 쓰일까?

- 기능 개발은 `feature/*`
- 통합은 `main`
- 실제 배포는 `staging`, `production` 같은 환경 브랜치로 관리

즉, 브랜치 자체가 배포 파이프라인의 흐름을 보여준다.

### 장점은?

- 개발/운영 환경 흐름이 명확하다
- CI/CD와 자연스럽게 연결된다
- Git Flow보다 가볍고, GitHub Flow보다 안정적이다

### 단점은?

- 환경 브랜치 관리가 필요하다
- 팀마다 규칙을 잘 정하지 않으면 혼란스러울 수 있다

### 이런 프로젝트에 잘 맞는다

- CI/CD를 적극적으로 사용하는 팀
- Dev / Prod 환경 분리가 중요한 서비스
- 클라우드, 컨테이너 기반 프로젝트

## 4. Trunk-Based Development — 자동화를 전제로 한 전략

Trunk-Based Development는 접근 방식부터 다르다.  
“브랜치를 나누는 게 문제의 원인 아닐까?”라는 질문에서 출발한다.

이 전략에서는 하나의 중심 브랜치(`main` 또는 `trunk`)를 중심으로,  
아주 짧은 브랜치만 쓰거나 아예 브랜치를 만들지 않는다.

### 대신 무엇이 중요할까?

- 강력한 자동 테스트
- CI 파이프라인
- 기능 토글(feature toggle)

사람이 아니라 **자동화가 안정성을 책임진다.**

### 장점은?

- 병합 충돌이 거의 없다
- 개발 속도가 매우 빠르다
- 지속적 통합에 최적화돼 있다

### 단점은?

- 테스트가 부실하면 바로 사고로 이어진다
- 팀의 성숙도가 낮으면 위험하다

### 이런 프로젝트에 잘 맞는다

- 대규모 조직
- 자동화가 잘 갖춰진 팀
- 마이크로서비스 환경

## 5. Release Branch 전략 — 여러 버전을 동시에 관리해야 할 때

이 전략은 “항상 최신만 배포”하는 서비스보다는,  
**여러 버전을 동시에 유지해야 하는 제품**에서 주로 사용된다.

예를 들어 고객사마다 다른 버전을 쓰거나,  
LTS(Long Term Support)를 제공해야 하는 경우다.

### 특징적인 흐름

- `main`은 기준이 되는 최신 코드
- `release/x.y` 브랜치로 버전별 관리
- 버그 수정은 필요한 릴리즈 브랜치에만 반영

### 장점은?

- 다중 버전 유지보수가 가능하다
- 고객 대응이 유연하다

### 단점은?

- 브랜치 관리 부담이 크다
- 유지보수 비용이 높다

### 이런 프로젝트에 잘 맞는다

- 엔터프라이즈 소프트웨어
- 장기 지원이 필요한 제품
- 고객사별 커스터마이징이 많은 서비스

## 마무리하며

Git Branch 전략은 “이게 정답이다”라고 말할 수 있는 문제가 아니다.  
중요한 건 **우리 팀이 어떤 방식으로 일하고,  
어떤 가치를 더 중요하게 생각하느냐**다.

- 안정성이 중요하면 Git Flow
- 속도와 단순함이 중요하면 GitHub Flow
- 배포 환경이 핵심이면 GitLab Flow
- 자동화가 강하면 Trunk-Based
- 다중 버전이면 Release Branch

브랜치 전략은 도구가 아니라 문화에 가깝다.  
팀에 맞게 선택하고, 필요하면 언제든 바꿀 수 있다는 점을 기억하면 좋겠다.
