# Harness Engineering이란? AI Agent를 제대로 일하게 만드는 방법

AI Agent를 써보면 처음에는 꽤 놀랍다.  
간단한 기능을 부탁하면 코드를 만들고, 테스트를 고치고, 문서까지 어느 정도 정리해준다.

그런데 조금만 일이 커지면 문제가 생긴다.

- Agent가 프로젝트 구조를 제대로 이해하지 못한다.
- 이미 정해둔 규칙과 다른 방식으로 코드를 만든다.
- 테스트는 통과했지만 실제 동작은 이상하다.
- 같은 실수를 다음 작업에서도 반복한다.
- 문서, 코드, 설정이 서로 어긋나기 시작한다.

이때 단순히 "프롬프트를 더 잘 써야 하나?"라고 생각하기 쉽다.  
하지만 최근 AI 개발 흐름에서는 문제를 조금 다르게 본다.

**문제는 모델 하나가 아니라, 모델을 둘러싼 작업 환경 전체에 있다.**

이 작업 환경을 설계하는 분야를 요즘 **Harness Engineering**이라고 부른다.

## Harness Engineering은 무엇인가?

Harness Engineering은 AI 모델이나 Agent가 안정적으로 일할 수 있도록  
**제약, 문맥, 도구, 검증, 피드백 루프를 설계하는 엔지니어링**이다.

간단히 말하면 아래와 같다.

> Model만 있으면 그냥 똑똑한 엔진이다.  
> Harness가 있어야 실제로 일을 끝내는 Agent가 된다.

그래서 최근에는 이런 식으로 설명하기도 한다.

```text
Agent = Model + Harness
```

여기서 Model은 GPT, Claude 같은 언어 모델이다.  
Harness는 그 모델이 일하는 방식 전체를 감싸는 시스템이다.

- 어떤 정보를 읽을 수 있는가?
- 어떤 파일을 수정할 수 있는가?
- 어떤 규칙은 반드시 지켜야 하는가?
- 결과가 맞는지 어떻게 검증하는가?
- 실패했을 때 어떻게 다시 고치게 할 것인가?

이 질문들에 답하는 구조를 만드는 것이 Harness Engineering이다.

## 왜 갑자기 중요해졌을까?

예전에는 AI를 주로 질문 답변이나 짧은 코드 작성에 사용했다.  
이때는 프롬프트만 잘 써도 꽤 괜찮은 결과가 나왔다.

하지만 지금은 AI Agent가 더 긴 작업을 맡기 시작했다.

- 여러 파일을 동시에 수정한다.
- 테스트를 실행하고 실패 원인을 찾는다.
- PR을 만들고 리뷰에 대응한다.
- 브라우저를 열어 실제 UI를 확인한다.
- 로그와 메트릭을 읽고 장애를 추적한다.

이런 단계로 오면 "말을 잘 거는 기술"만으로는 부족하다.  
Agent가 오래 일할수록 방향을 잃거나, 잘못된 패턴을 반복하거나, 프로젝트의 암묵적인 규칙을 놓치기 쉽기 때문이다.

OpenAI의 Codex 팀은 2026년 2월에 Harness Engineering에 관한 글을 공개했다.  
그 글에서 팀은 사람이 직접 코드를 작성하지 않고 Codex Agent들로 내부 제품을 만들면서 배운 점을 정리했다. 핵심은 사람이 더 이상 모든 코드를 직접 쓰는 것이 아니라, Agent가 안정적으로 일할 수 있는 환경을 설계해야 한다는 점이었다.

즉, 앞으로의 엔지니어링은 단순히 "AI에게 일을 시키는 것"이 아니라  
**AI가 좋은 결과를 낼 수밖에 없는 구조를 만드는 것**에 가까워지고 있다.

## 1. Context Engineering: Agent에게 무엇을 보여줄 것인가?

AI Agent는 사람이 아니다.  
팀 분위기, 과거 회의 내용, 암묵적인 컨벤션, "우리 팀은 원래 이렇게 해요" 같은 정보를 자연스럽게 알지 못한다.

Agent에게 보이지 않는 정보는 사실상 존재하지 않는 정보다.

그래서 Harness Engineering에서 가장 먼저 중요한 것은 **문맥 관리**다.

### 대표적인 방법

- `README.md`에 프로젝트의 목적과 실행 방법을 명확히 적는다.
- `AGENTS.md` 같은 파일에 Agent가 따라야 할 작업 규칙을 둔다.
- 아키텍처 문서, 브랜치 전략, 테스트 전략을 저장소 안에 둔다.
- API 스키마, DB 스키마, 디자인 시스템 규칙을 Agent가 읽을 수 있게 만든다.
- 로그, 메트릭, 브라우저 상태 같은 동적 정보도 Agent가 접근할 수 있게 한다.

중요한 것은 모든 내용을 한 파일에 몰아넣는 것이 아니다.  
오히려 너무 큰 문서는 Agent에게 방해가 된다.

좋은 방식은 짧은 진입점과 잘 나뉜 문서를 두는 것이다.

```text
AGENTS.md
README.md
docs/
  architecture.md
  branch-strategy.md
  testing.md
  product-specs/
```

Agent는 먼저 작은 지도를 보고, 필요할 때 더 깊은 문서를 찾아가면 된다.

## 2. Architectural Constraints: 하지 말아야 할 일을 막는 방법

사람 개발자는 코드를 보다가 이상한 냄새를 느낀다.  
"이 레이어에서 저 레이어를 직접 부르면 안 되는데", "여기서 DB를 바로 만지면 안 되는데" 같은 감각이 있다.

AI Agent에게도 이런 감각을 기대할 수는 있지만, 그것만 믿기는 어렵다.

그래서 Harness에서는 규칙을 문서로만 두지 않고, 가능하면 **기계적으로 강제**한다.

### 예시

- ESLint로 import 경로 제한하기
- 테스트로 계층 간 의존 방향 검증하기
- 타입 체크로 데이터 구조 강제하기
- formatter로 코드 스타일 통일하기
- commit hook이나 CI에서 규칙 위반을 막기
- 파일 크기, 함수 크기, 네이밍 규칙을 검사하기

이런 제약은 사람에게는 조금 답답해 보일 수 있다.  
하지만 Agent에게는 오히려 길을 만들어준다.

규칙이 명확하면 Agent는 선택지를 덜 헤매고, 잘못된 방향으로 멀리 가기 전에 멈춘다.

## 3. Feedback Loop: 실패를 다음 성공의 재료로 만드는 방법

AI Agent가 한 번에 완벽한 결과를 내기는 어렵다.  
중요한 것은 실패했을 때 그 실패가 그냥 끝나지 않도록 만드는 것이다.

Harness Engineering에서는 실패를 피드백으로 바꾸는 루프가 중요하다.

### 기본적인 루프

```text
작업 지시
→ 코드 수정
→ 테스트 실행
→ 실패 원인 분석
→ 수정
→ 다시 검증
```

여기까지는 일반적인 개발 흐름과 비슷하다.

하지만 Agent 시대에는 이 루프를 더 적극적으로 자동화할 수 있다.

- 테스트 실패 로그를 Agent가 직접 읽게 한다.
- 브라우저 스크린샷이나 DOM 상태를 Agent가 확인하게 한다.
- PR 리뷰를 다른 Agent가 수행하게 한다.
- 반복되는 리뷰 코멘트는 문서나 lint 규칙으로 승격한다.
- 장애 로그를 기반으로 재현 절차를 만들게 한다.

좋은 Harness는 Agent가 틀리지 않게만 하는 시스템이 아니다.  
틀렸을 때 더 나은 방향으로 돌아오게 만드는 시스템이다.

## 4. Garbage Collection: Agent가 만든 부채를 정리하는 방법

Agent는 기존 패턴을 잘 따라 한다.  
이 장점은 동시에 위험이기도 하다.

프로젝트 안에 애매한 코드가 있으면 Agent는 그 애매한 코드를 정상 패턴으로 착각할 수 있다.  
문서가 오래되어 있으면 오래된 문서를 기준으로 새 코드를 만들 수 있다.

그래서 Harness Engineering에서는 주기적인 정리가 필요하다.

### 정리 대상

- 실제 코드와 맞지 않는 문서
- 더 이상 쓰지 않는 실행 계획
- 반복되는 나쁜 코드 패턴
- 중복된 유틸 함수
- 깨진 링크나 오래된 명령어
- 테스트 없이 남은 중요한 로직

이 작업을 OpenAI 글에서는 일종의 garbage collection처럼 설명한다.  
메모리에서 쓸모없는 객체를 정리하듯, 코드베이스에서도 Agent가 계속 일할 수 있도록 낡은 맥락과 나쁜 패턴을 정리하는 것이다.

## 5. Harness Engineering과 Prompt Engineering의 차이

Prompt Engineering은 여전히 중요하다.  
하지만 그것만으로는 충분하지 않다.

차이를 간단히 정리하면 다음과 같다.

| 구분 | 관심사 |
| --- | --- |
| Prompt Engineering | 모델에게 어떻게 말할 것인가 |
| Context Engineering | 모델에게 무엇을 보여줄 것인가 |
| Harness Engineering | 모델이 어떤 환경과 규칙 안에서 일하게 할 것인가 |

프롬프트가 대화의 기술이라면,  
컨텍스트는 자료를 배치하는 기술이고,  
Harness는 작업장을 설계하는 기술에 가깝다.

좋은 프롬프트는 한 번의 답변을 좋게 만들 수 있다.  
좋은 Harness는 반복되는 작업 전체의 품질을 끌어올린다.

## 6. 우리 프로젝트에 적용한다면?

이 저장소를 기준으로 생각하면 Harness Engineering은 아주 거창한 시스템부터 시작할 필요가 없다.

작게 시작한다면 이런 것부터 할 수 있다.

- `README.md`에 프로젝트 목적과 문서 목록을 정리한다.
- `Git_Branch_Strategy.md`처럼 팀의 작업 규칙을 문서화한다.
- Agent가 참고할 `AGENTS.md`를 만든다.
- 테스트 실행 명령과 검증 방법을 명확히 적는다.
- PR 전에 확인해야 할 체크리스트를 둔다.
- 반복되는 실수는 문서가 아니라 lint나 script로 막는다.

조금 더 나아가면 이런 것도 가능하다.

- 브랜치 이름 규칙 검사
- 커밋 메시지 규칙 검사
- 문서 링크 유효성 검사
- 테스트 커버리지 기준 설정
- 아키텍처 계층 위반 검사

처음부터 완벽한 Harness를 만들 필요는 없다.  
Agent가 헷갈리는 지점이 발견될 때마다 하나씩 규칙과 도구로 바꾸면 된다.

## 장점

- Agent가 프로젝트 규칙을 더 잘 따른다.
- 사람이 반복적으로 설명해야 하는 일이 줄어든다.
- 테스트, 문서, 아키텍처가 함께 관리된다.
- 실수가 한 번의 수정으로 끝나지 않고 시스템 개선으로 이어진다.
- 코드베이스가 사람뿐 아니라 Agent에게도 읽기 쉬워진다.

## 단점

- 처음에는 문서와 규칙을 정리하는 비용이 든다.
- 너무 많은 제약은 오히려 작업 속도를 늦출 수 있다.
- 잘못 만든 규칙은 Agent의 좋은 선택까지 막을 수 있다.
- Harness도 코드처럼 계속 관리하지 않으면 금방 낡는다.

## 언제 특히 필요할까?

Harness Engineering은 모든 프로젝트에 같은 수준으로 필요한 것은 아니다.  
하지만 아래 상황에서는 중요도가 빠르게 올라간다.

- AI Agent에게 실제 코드 수정을 자주 맡기는 경우
- 프로젝트 규모가 커지고 문서가 흩어지는 경우
- 여러 명이 같은 코드베이스에서 빠르게 작업하는 경우
- 테스트, 배포, 리뷰 자동화가 중요한 경우
- 장기적으로 Agent 기반 개발 워크플로우를 만들고 싶은 경우

특히 Agent가 단순 보조 도구를 넘어 "작업자"처럼 움직이기 시작하면,  
Harness는 선택이 아니라 기반에 가까워진다.

## 정리

Harness Engineering은 새로운 유행어처럼 보일 수 있다.  
하지만 그 안에 있는 문제의식은 꽤 현실적이다.

AI 모델은 점점 강해지고 있다.  
하지만 강한 모델이 곧 안정적인 결과를 의미하지는 않는다.

안정적인 결과를 만들려면 모델 주변에 좋은 작업 환경이 필요하다.

- 읽을 수 있는 문서
- 지켜야 할 제약
- 실행 가능한 테스트
- 실패를 고치는 피드백 루프
- 오래된 맥락을 정리하는 유지보수

이 모든 것을 설계하는 일이 Harness Engineering이다.

앞으로 개발자의 역할은 코드를 직접 쓰는 일에서 완전히 사라지지는 않겠지만,  
점점 더 **Agent가 좋은 코드를 만들 수 있는 시스템을 설계하는 일**로 확장될 것이다.

결국 중요한 질문은 이것이다.

> AI에게 어떤 일을 시킬 것인가?  
> 그리고 그 AI가 제대로 일할 수밖에 없는 환경을 어떻게 만들 것인가?
